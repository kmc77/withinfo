## 1. 행추가 시 동일 기업 최후 데이터 자동 복사 구현

### 요구사항

행추가 후 기업 검색 시, 같은 기업의 직전 row 데이터가 있으면 가져와서 복사. 사업상태는 단계 진행(신규→연장→종료). 진행상태는 "진행"으로 초기화.

### 설계 방향

- 기존 쿼리 ID(`selectLoanIntSprtBizList`)에 **`<if>` 분기만 추가**, 신규 쿼리 ID 만들지 않음
- 화면단에 **행추가용 별도 dataList + submission** 추가 (그리드 dataList 건드리지 않음)

### 1-A. SQL 분기 추가

기존 `selectLoanIntSprtBizList`의 WHERE 절 끝에 `clcmNo` 분기 추가:

```xml
<if test='clcmNo != "" and clcmNo != null'>
AND A.CLCM_NO = #{clcmNo}
AND A.MNG_NO = (
    SELECT MNG_NO FROM (
        SELECT MNG_NO
        FROM TPS_LOAN_INT_SPRT_BIZ_B
        WHERE CLCM_NO = #{clcmNo}
        ORDER BY FRST_REG_DT DESC, MNG_NO DESC
    )
    WHERE ROWNUM = 1
)
</if>
```

**동작 분기:**

| 호출 | clcmNo | 결과 |
|---|---|---|
| 목록 조회 (기존) | 빈값 | if 안 탐, 전체 조회 그대로 |
| 행추가 트리거 (신규) | 있음 | 해당 기업 최신 1건 |

### 1-B. 정렬 키 결정 — MAX 1건 보장 함정

처음엔 `MAX(FRST_REG_DT)` 서브쿼리 단독으로 시작. D00141 케이스 테스트 → **결과 2행** 반환.

원인 분석:

```sql
SELECT FRST_REG_DT, COUNT(*), MIN(MNG_NO), MAX(MNG_NO)
FROM TPS_LOAN_INT_SPRT_BIZ_B
WHERE CLCM_NO = 'D00141'
GROUP BY FRST_REG_DT
HAVING COUNT(*) > 1;
```

같은 `FRST_REG_DT` 시각에 INSERT된 row가 4쌍 (총 8건) 존재. trigger/batch로 동시 등록된 추정. → MAX(FRST_REG_DT) 단독으로는 1건 보장 불가.

**해결:** 2차 정렬키로 PK인 MNG_NO 추가.

```sql
ORDER BY FRST_REG_DT DESC, MNG_NO DESC
+ ROWNUM = 1
```

### 1-C. ROWNUM=1 패턴 주의

Tibero/Oracle 공통 함정: `WHERE ROWNUM=1 ORDER BY ...` 쓰면 **정렬 전에 ROWNUM 적용** → 엉뚱한 1건.

**반드시 서브쿼리로 감싸기:**

```sql
SELECT MNG_NO FROM (
    SELECT MNG_NO FROM ... ORDER BY ...
)
WHERE ROWNUM = 1
```

### 1-D. 백엔드 ReqVO 필드 추가

기존 `LoanIntSprtBizReqVO`엔 `mngNo`, `prgrsStClssCd`, `cmlBgngYmd`, `cmlEndYmd`만 있었음.

**`clcmNo` 필드 + getter/setter 추가 필수.** 안 하면 프론트에서 set해도 서버 단에서 null로 받음 → 분기 안 탐.

```java

public void setClcmNo(String clcmNo) { this.clcmNo = clcmNo; }
```

## 2. 트러블슈팅 기록

### 2-A. submit 호출 시 TypeError

증상: `sbm_selectLastLoanIntSprtBiz.submit()` 타입에러
해결: WebSquare 환경에선 `com.sbm.execute(sbm_xxx)` 사용

### 2-B. 페이로드에 clcmNo 안 박힘

증상: 호출은 가는데 payload에 clcmNo 빠짐 → 백엔드 분기 안 탐

원인 후보 2가지:
- 응답 VO(`dma_lastLoanIntSprtBizVO`)에 set했음. **요청 VO**(`dma_loanIntSprtBizReqVO`)에 박아야 함
- 요청 dataMap에 `<w2:key id="clcmNo"/>` 키 누락

해결: 요청 VO에 set + key 정의 확인

### 2-C. 백엔드 ReqVO 필드 누락

증상: payload엔 들어가는데 SQL 파라미터로 null로 전달
원인: 백엔드 VO에 필드 미정의 → Spring 바인딩 단계에서 무시
해결: `private String clcmNo;` + getter/setter 추가

### 2-D. 응답 빈 배열 (dataMap 매핑 실패)

증상: Network는 정상 응답, dataMap엔 빈값
원인: 응답 JSON 키 `loanIntSprtBizListVO`와 dataMap ID 불일치
해결: 
1. submission target에 `"key":"loanIntSprtBizListVO"` 명시
2. 응답이 List 형태 `[{...}]`라 dataMap이 아닌 **dataList로 그릇 변경**

### 2-E. 그리드 데이터 날아가는 문제

증상: 기업 선택 후 그리드의 기존 row들 사라지고 신규 1줄만 남음
원인: target dataList가 그리드 바인딩된 `dlt_loanIntSprtBizListVO`로 잘못 잡힘
해결: 신규 `dlt_lastLoanIntSprtBizVO` 만들고 그쪽으로 target 변경

### 2-F. setCellData 후 화면 갱신 안 됨 (가장 어려웠던 케이스)

증상: 콘솔 찍어보면 src에 값 다 있는데 화면엔 일부 컬럼만 표시

디버깅 과정:
1. `dl.getColumnIndex` 확인 → 정상 인덱스 반환
2. row 인덱스 확인 → `getRowPosition() + 1 = 263`
3. `dl.getRowCount() === 263` 확인 → **마지막 row 인덱스는 262**
4. row 263 위치엔 빈 객체만 존재 (WebSquare가 인덱스 초과 접근 시 빈 객체 반환)

원인: **`+1` 잘못 적용** — `callbackClcmNm`은 row 262에 박는데 submitdone에선 row 263에 박으려 시도 → 존재 않는 row → setCellData 무시

해결: submitdone에서 `+1` 제거. `getRowPosition()` 그대로 사용

### 2-G. clcmNo 컬럼 dataList 누락으로 isEmpty 분기 다 잡힘

증상: src에 값 있는데도 `isEmpty(src.getCellData(0, "clcmNo"))`가 true → return
원인: `dlt_lastLoanIntSprtBizVO`의 columnInfo에 `clcmNo` 누락
해결: 비었음 체크를 `clcmNm`(확실히 정의된 컬럼)으로 변경

### 2-H. 다른 기업 데이터로 혼동

증상: DBeaver 결과(2023-07-26)와 화면 응답(2023-07-21) 달라 보임
원인: DBeaver는 D00099, 화면 테스트는 D00141 → 서로 다른 기업
교훈: 디버깅 시 같은 기업코드로 비교 필수

## 3. 파일등록 버튼 readOnly 클릭 차단 안 되는 문제 (해결)

### 증상

`scwin.setGrdReadOnly(true)` 호출 시 파일등록 버튼 컬럼이 **시각적으로 회색 처리되긴 하는데 클릭하면 함수가 실행됨**.

### 원인

**버튼 컬럼은 `setColumnReadOnly` / `setCellInputType readOnly`로 클릭 못 막음.**

4/23 세션에서 정립한 원칙 재확인:
- `setDisabled` → button 컬럼 제어에 효과적 (회색 + 클릭 차단)
- `setCellInputType readOnly` → text/calendar 컬럼 전용
- 버튼 컬럼에 readOnly 적용 시 → 시각적 회색만 적용되고 onclick 이벤트는 살아있음

CSS:
```css
/* disabled와 readonly 둘 다 동일한 회색 스타일 */
.w2grid_default_disabled button,
.w2grid_default_readonly button {
    border: 1px solid #cdd0d3;
    cursor: auto;
    color: #999;
    background: #e7e7e7;
}
```

→ 시각 동일, 동작 다름

### 해결

파일 버튼 컬럼 4개만 `setDisabled("column", ...)` 분기:

```javascript
scwin.setGrdReadOnly = function(obj) {
    if (obj == true) {
        grd_loanIntSprtBizListVO.setGridReadOnly(true);
        
        // 일반 컬럼은 setColumnReadOnly (기존 그대로)
        grd_loanIntSprtBizListVO.setColumnReadOnly("_chk", true);
        grd_loanIntSprtBizListVO.setColumnReadOnly("clcmNm", true);
        // ... 나머지 일반 컬럼들 ...
        
        // 파일 버튼 컬럼은 setDisabled로 (readOnly는 클릭 못 막음)
        grd_loanIntSprtBizListVO.setDisabled("column", "acinRpppFileGrpId",         true);
        grd_loanIntSprtBizListVO.setDisabled("column", "rmmdFileGrpId",             true);
        grd_loanIntSprtBizListVO.setDisabled("column", "sprtTrgtEvdcPprsFileGrpId", true);
        grd_loanIntSprtBizListVO.setDisabled("column", "loanNtshFileGrpId",         true);
        
    } else if (obj == false) {
        // false 분기도 동일 구조
        grd_loanIntSprtBizListVO.setDisabled("column", "acinRpppFileGrpId",         false);
        grd_loanIntSprtBizListVO.setDisabled("column", "rmmdFileGrpId",             false);
        grd_loanIntSprtBizListVO.setDisabled("column", "sprtTrgtEvdcPprsFileGrpId", false);
        grd_loanIntSprtBizListVO.setDisabled("column", "loanNtshFileGrpId",         false);
    }
};
```

## 4. 만기일 자동계산 로직 설계 (정책 확정, 구현 대기)

### 화면설계서 룰

| 상태 | 만기일 계산 |
|---|---|
| 신규 | 실행일(최초) + 1년 |
| 1차 연장 | 실행일(최초) + 2년 |
| 2차 연장 | 실행일(최초) + 3년 |

### 정책 분석

3안 트레이드오프:

| 안 | 장점 | 단점 |
|---|---|---|
| 자동만 (수정 불가) | 일관성 100% | 예외 계약 처리 못 함 |
| 자동+수동 허용 | 기본 표준 + 예외 가능 | 데이터 출처 불명 |
| 수동만 | 자유도 최대 | 매번 사람이 계산, 실수 빈번 |

**확정: 자동계산 기본 + 수동 덮어쓰기 허용**

### 트리거 위치 비교

| 위치 | 동작 |
|---|---|
| `oncellendedit` 유효성 함수 | 상태/실행일 변경 시 자동 재계산. 만기일 수동 입력은 cellendedit에서 자동계산 호출 안 함 → 보존 |
| `submitdone` (행추가) | 행추가 직후 1회만 |

**채택: `oncellendedit`** — 단 사용자가 만기일 수동 입력 후 상태/실행일 또 바꾸면 자동값으로 덮어써지는 트레이드오프 인지.

```javascript
    var addYear = bizStCd === "31420001" ? 1

                : bizStCd === "31420003" ? 3
                : 0;
    
    if (addYear === 0) return;
    
    var mtryYmd = com.date.addYear(excnYmd, addYear);
    dl.setCellData(rowIndex, "mtryYmd", mtryYmd);
};
```

## 5. 핵심 교훈

| 구분 | 내용 |
|---|---|
| 쿼리 분기 재사용 | 신규 쿼리 ID 만들지 말고 기존 쿼리에 `<if>` 분기로 추가. 호출 시 다른 dataList target으로 분리하면 충돌 없음 |
| ROWNUM=1 패턴 | 반드시 서브쿼리로 감싸기. `WHERE ROWNUM=1 ORDER BY` 패턴 금지 (정렬 전 ROWNUM 적용됨) |
| MAX 1건 보장 | 같은 시각 다건 가능 시 2차 정렬키(PK) 추가 필수. DATE 컬럼만으론 unique 보장 안 됨 |
| 백엔드 VO 필드 추가 | 프론트에서 set 해도 백엔드 VO에 필드 없으면 null 바인딩. 분기 if문 안 탐 |
| 응답 키 매핑 | dataList ID와 응답 JSON 키 불일치 시 자동 매핑 실패. submission target에 `"key":"..."` 명시 |
| dataMap vs dataList | 응답이 단건이라도 `[{...}]` List 형태로 오면 dataList로 받아야 함 |
| getRowPosition() +1 함정 | `getRowCount()` 초과 row는 빈 객체로 존재하는 척만 함. setCellData 무시. 유효 인덱스는 `getRowCount() - 1`까지 |
| columnInfo 누락 | dataList columnInfo에 없는 컬럼은 응답에 와도 무시. 비었음 체크는 확실히 정의된 컬럼으로 |
| readOnly vs disabled | **button 컬럼은 setDisabled로만 클릭 차단.** readOnly는 시각만 회색. CSS는 동일하지만 동작 다름 |
| 자동계산 + 수동 입력 정책 | 트리거 시점 분리 (자동: 상태/실행일, 수동: 만기일). 본질적 트레이드오프 인지 |
