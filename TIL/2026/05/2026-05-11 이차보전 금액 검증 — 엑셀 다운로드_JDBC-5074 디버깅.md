## 4. 그리드 세로 스크롤 2개 표시 문제

### 증상

이자보전금 상세 그리드에 세로 스크롤바가 2개 표시됨.

### 원인 분석

XML 확인 결과:

```xml
<w2:tabControl id="teb" ...>
    <w2:tabs id="tabs2" .../>
    <w2:tabs id="tabs3" .../>
    <w2:content alwaysDraw="false" style="height:654px;" id="content2">  <!-- 부모 654 -->
        <w2:gridView id="grd_IdstAmtCmpt"
                     style="height: 725px;"                                 <!-- 자식 725 -->
                     ...>
```

부모(`<w2:content>`)는 `654px`인데 자식 그리드는 `725px`. 자식이 부모보다 71px 큼 → 부모에서 스크롤 1개, 그리드 자체에서 스크롤 1개 = 총 2개.

### 해결

그리드 height를 부모 이하로 변경:

```xml
<w2:gridView style="height: 654px;" ...>
```

또는 부모 100% 채우기:

```xml
<w2:gridView style="height: 100%;" ...>
```

### 부수 메모

`<w2:caption value="this is a grid caption.">` 디폴트 문구가 그대로 남아있음. 추후 정리.

### 상태

수정 안내 완료.

## 5. JDBC-5074 조회 에러 — 가장 큰 이슈

### 에러 메시지

```
java.sql.SQLException: JDBC-5074: Given string does not represent a number in proper format.
```

### 증상

기준년월 값에 따라 결과가 다름:

| 기준년월 | 결과 |
|---------|------|
| 202401 | 에러 |
| 202402 | 정상 |
| 202403 | 에러 |
| 202605 | 정상 (기본 테스트값) |

### 1차 진단

서버 로그에 SQL 출력됨. 매퍼 위치:

```
kobc.ngs.sf.ps.csbs.mapper.IdstAmtCmptMapper.selectIdstAmtCmptList
```

쿼리 구조 (대략):

```sql
WITH MAX_AA AS ( /* SQL_ID: ... */
    SELECT ... 
)
, AA AS (
    SELECT A.APLY_NO
         , B.FNNC_WTHD_TRNM
         , B.GRNF_CMPT_CLSS_CD
         , NVL(E.THBL_AMT, 0)            AS THBL_AMT
         , NVL(F.RPMT_PRIN_AMT, 0)       AS RPMT_PRIN_AMT
         , TO_CHAR(TRUNC(TO_DATE('202605','YYYYMM')), 'YYYY-MM-DD') S_YMD
         , CASE WHEN F.THBL_AMT != 0 THEN F.RPMT_YMD
                ELSE ''
           END NOW_RPMT_YMD
         , F.RPMT_YMD AS FST_RPMT_YMD
         , G.RPMT_YMD AS SEC_RPMT_YMD
         ...
         , CASE WHEN F.RPMT_YMD IS NOT NULL THEN 
                    CASE WHEN F.THBL_AMT != 0 THEN 
                        F.RPMT_YMD - TO_CHAR(TRUNC(TO_DATE('202605','YYYYMM')),'YYYYMMDD') + 1
                        ELSE ''
                    END
                ELSE 0
           END FST_CNT
         , CASE WHEN F.RPMT_YMD IS NOT NULL THEN 
                    CASE WHEN G.RPMT_YMD IS NOT NULL THEN G.RPMT_YMD - F.RPMT_YMD
                        ELSE TO_CHAR(LAST_DAY(...), 'YYYYMMDD') - F.RPMT_YMD
                    END
                ELSE 0
           END SEC_CNT
         ...
    FROM (...) A
       , (...) B
       , TGU_GRNF_CMPT_I C
       , TGU_ASSR_APLY_I E
       , TGU_FNNC_CTRT_RPMT_SCHD_I F
       , TGU_FNNC_CTRT_RPMT_SCHD_I G
    WHERE A.APLY_NO = B.APLY_NO
      ...
      AND #{crtrYm} = G.CRTR_YM(+)
      AND F.BFTH_LTWR_NO IS NOT NULL
)
SELECT ...
ORDER BY F.BFTH_LTWR_NO
```

### 스택트레이스 핵심

```
at com.tmax.tibero.jdbc.driver.TbPreparedStatementImpl.executeInternal
at com.tmax.tibero.jdbc.driver.TbPreparedStatementImpl.execute
at net.sf.log4jdbc.sql.jdbcapi.PreparedStatementSpy.execute
at org.apache.ibatis.executor.statement.PreparedStatementHandler.query
```

> `executeInternal` 단계 = **DB가 쿼리 실행하다가** 터짐. 결과 반환 전. MyBatis 매핑/VO는 무관.

### 검토한 가설 전체

#### 가설 1. 회차 컬럼(`FNNC_CTRT_RPMT_TRNM`)에 비숫자 값 섞임

`MAX(E.FNNC_CTRT_RPMT_TRNM)` 부분에서 MAX 함수가 숫자처럼 다루다 터질 가능성.

확인 쿼리:
```sql
SELECT FNNC_CTRT_RPMT_TRNM
  FROM TGU_FNNC_CTRT_RPMT_SCHD_I
 WHERE FNNC_CTRT_RPMT_TRNM IS NOT NULL
   AND NOT REGEXP_LIKE(FNNC_CTRT_RPMT_TRNM, '^[0-9]+$');
```

→ **0건. 기각.**

#### 가설 2. `RPMT_YMD` 컬럼에 하이픈/공백 섞임

쿼리 분석 결과 `E.RPMT_YMD(+) < TRUNC(TO_DATE('202605','YYYYMM'))` 비교에서 String을 날짜로 암묵 변환 시도 의심.

확인 쿼리:
```sql
SELECT RPMT_YMD, LENGTH(RPMT_YMD)
  FROM TGU_FNNC_CTRT_RPMT_SCHD_I
 WHERE RPMT_YMD IS NOT NULL
   AND NOT REGEXP_LIKE(RPMT_YMD, '^[0-9]{8}$');
```

→ **0건. 기각.**

#### 가설 3. CASE 표현식 `ELSE ''` vs 숫자 분기 타입 충돌

```sql
CASE WHEN F.RPMT_YMD IS NOT NULL THEN 
    CASE WHEN F.THBL_AMT != 0 THEN F.RPMT_YMD - TO_CHAR(...) + 1  -- 숫자
         ELSE ''                                                    -- 문자열
    END
    ELSE 0                                                          -- 숫자
END FST_CNT
```

CASE 안에 숫자(`... + 1`, `0`)와 빈 문자열(`''`)이 섞임. Tibero가 타입 통일하려고 `''`를 숫자로 변환 시도 → 실패 가설.

`F.THBL_AMT = 0`인 1월 레코드가 처음 발견되면서 `ELSE ''` 분기 발동 가설.

확인 쿼리:
```sql
SELECT COUNT(*)
  FROM TGU_FNNC_CTRT_RPMT_SCHD_I F
 WHERE F.RPMT_YMD IS NOT NULL
   AND F.THBL_AMT = 0;
```

대응: `ELSE ''` → `ELSE 0`으로 변경, 그리고 빼기 연산도 0으로 치환.

→ **여전히 에러. 기각.**

#### 가설 4. VO 타입 불일치

서버단에서 VO 필드 타입(String/BigDecimal/Integer) 매핑 문제 의심.

→ **기각.** 근거 2가지:
1. DBeaver에서 raw SQL 실행해도 동일 에러
2. 스택트레이스 `executeInternal` 단계에서 터짐. VO 매핑은 결과 받은 후이므로 여기까지 도달 못 함.

### 최종 유력 원인 (사수 컨펌 받아야 함)

쿼리 안에서 **String 컬럼끼리 또는 `TO_CHAR` 결과와 빼기 연산**:

```sql
F.RPMT_YMD - TO_CHAR(TRUNC(TO_DATE('202605','YYYYMM')),'YYYYMMDD') + 1
G.RPMT_YMD - F.RPMT_YMD
TO_CHAR(LAST_DAY(...), 'YYYYMMDD') - F.RPMT_YMD
TO_CHAR(LAST_DAY(...), 'YYYYMMDD') - G.RPMT_YMD
```

`RPMT_YMD`는 String 컬럼. Tibero는 String 빼기 만나면 양쪽을 `TO_NUMBER`로 암묵 변환. 통상 변환 성공하지만, 특정 월 데이터 조합에서 변환 실패 발생.

### 추가 문제: 로직 오류

빼기 연산 자체가 보전일수를 **숫자 차이**로 계산함. 같은 달이면 우연히 맞지만:

```
20240131 - 20240101 = 30  (실제 30일, OK)
20240201 - 20240131 = 70  (실제 1일, 오류)
```

월 경계 넘으면 결과 틀어짐.

### 제안 해결책 (정석)

```sql
-- AS-IS
F.RPMT_YMD - TO_CHAR(TRUNC(TO_DATE('202605','YYYYMM')),'YYYYMMDD') + 1
-- TO-BE
TO_DATE(F.RPMT_YMD,'YYYYMMDD') - TRUNC(TO_DATE('202605','YYYYMM')) + 1

-- AS-IS
G.RPMT_YMD - F.RPMT_YMD
-- TO-BE
TO_DATE(G.RPMT_YMD,'YYYYMMDD') - TO_DATE(F.RPMT_YMD,'YYYYMMDD')

-- AS-IS
TO_CHAR(LAST_DAY(TO_DATE('202605','YYYYMM')), 'YYYYMMDD') - F.RPMT_YMD
-- TO-BE
LAST_DAY(TO_DATE('202605','YYYYMM')) - TO_DATE(F.RPMT_YMD,'YYYYMMDD')
```

`DATE - DATE = 숫자(일수)` 정상 반환. **부수 효과로 월 경계 안전.**

### 검증 원칙

> 한꺼번에 다 바꾸지 말고 빼기 1개씩 변환 → 조회 시도 → 통과/실패 기록.
> 그래야 어느 라인이 진짜 범인인지 확정 가능.
> 다 바꿔서 통과하면 진짜 원인 미궁.

### 상태

**검증 미완료.** 사수 컨펌 후 단계적 적용 필요.

## 6. `if test` 진행중사업 체크박스 조건 미동작

### 증상

```xml
<if test='prgrsYn == "Y"'>
    AND TO_CHAR(SYSDATE, 'YYYYMMDD') BETWEEN E.ASSR_BGNG_YMD
                                         AND E.ASSR_END_YMD
</if>
```

체크박스 체크/언체크 무관하게 항상 `if` 분기 진입 → 쿼리 추가됨.

### 의심 원인

1. **체크박스 unchecked 시 값 미초기화** — 컴포넌트가 `"Y"` 값을 그대로 보내거나 빈 값으로 안 바꿈
2. **체크박스 컴포넌트 `ref` 누락** — DataCollection에 값 안 들어가서 VO엔 항상 같은 값 매핑
3. **`trueValue`/`falseValue` 미설정** — 체크/언체크 값이 둘 다 `"Y"` 같이 동일해질 가능성

### 진단 방법

F12 Network → 조회 호출 → Payload 확인:
- 체크: `prgrsYn` 값
- 언체크: `prgrsYn` 값
- 두 값 비교하면 원인 즉시 보임

### 상태

미해결.

## 핵심 학습 정리

### Tibero / SQL

- **JDBC-5074** = String → Number 암묵 변환 실패
- 스택트레이스 `executeInternal`에서 터짐 = **쿼리 실행 단계** = VO 매핑 무관
- DBeaver raw SQL 재현이 진단 1차 수단
- 월별로 결과 다른 패턴 = 데이터 의존. 단, "어떤 데이터"는 가설 단계적 검증 필요
- String 컬럼 빼기 연산은 위험. `TO_DATE` 명시 변환이 정석
- 날짜 일수 차이: `DATE - DATE = 숫자(일수)`. String 빼기는 단순 숫자 차이라 월 경계에서 망함

### WebSquare

- 그리드 활성 탭 분기: `[부모탭컨트롤ID].getSelectedTabIndex()`
- 그리드 메서드에 `isVisible()` **없음**
- 탭컴포넌트에 `getSelectedIndex()` **미지원** → `getSelectedTabIndex()` 사용
- 개별 탭 시트 ID와 부모 탭컨트롤 ID 구분 필요
- 그리드 컬럼 공통코드 매핑: `compID:"그리드ID:컬럼ID"` 형식 (셀렉트박스와 별도)
- 부모/자식 height 불일치 → 스크롤 2개 발생
- 체크박스 `if test` 동작 안 함 → `ref`/`trueValue`/`falseValue` 점검

### Jenkins / Maven

- `cannot find symbol` 컴파일 에러 시 의심 순서:
  1. import 경로 오타
  2. 모듈 의존성 누락 (pom.xml)
  3. 클래스 실제 존재 여부 (`Ctrl+H` File Search)

### 잘못한 것

1. **데이터 의심 가설을 너무 오래 붙잡음**
   - "1월 X, 2월 O, 3월 X" 패턴 보고 즉시 데이터 깨짐 쪽으로 기울었음
   - 확인 쿼리 0건 나왔을 때 가설 빨리 버렸어야 했는데 다음 데이터 가설로 이동
   - 쿼리 구조 자체(암묵 변환 트리거)도 동일 증상 가능하다는 걸 늦게 깨달음

2. **VO 타입 의심에 잠깐 흔들림**
   - 스택트레이스 보면 `executeInternal`에서 터짐 = 쿼리 실행 단계
   - VO 매핑은 결과 받은 뒤 일어나는 거라 무관한데 잠깐 의심함

3. **이분탐색을 더 일찍 했어야 함**
   - 가설 2~3개 기각된 시점에서 SQL 자체 이분탐색으로 바로 전환했어야

4. **한꺼번에 수정 시도**
   - `ELSE ''` 다 바꾸고 빼기 다 0 처리 → 여전히 에러
   - 한 군데씩 단독 변경 → 결과 확인이 정석

### 다음에 적용할 원칙

- **JDBC 에러는 스택트레이스 위치로 단계부터 판단**
  - `executeInternal` / `prepareExecute` → DB 단
  - `ResultMapException` / `TypeException` → 매핑 단
  - `NumberFormatException` (java) → 애플리케이션 단

- **데이터 의존 패턴 봐도 쿼리 내 암묵 변환 지점 동시 확인**

- **가설 1개 검증에 5분 넘어가면 다음 가설 또는 이분탐색으로 전환**

- **수정은 한 군데씩, 결과 확인 후 다음**

- **사수 보고 시:** "TO_DATE 다 적용해서 됨"보다 "이 라인이 원인 확정됐고 이렇게 수정" 이 신뢰도 훨씬 높음
