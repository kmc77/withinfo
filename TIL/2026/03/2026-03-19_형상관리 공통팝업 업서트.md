## 2. 공통 팝업 개발 (고객정보 팝업)

### 팝업 타이틀 동적 세팅
호출하는 화면마다 팝업 제목이 달라야 하는 경우, `openPopup` 시 `opt.popupName`으로 전달

```javascript
// 호출하는 화면
var opt = {
    id: "H3P03",
    popupName: "고객정보",  // 화면마다 다르게
    width: "1200", height: "1220"
};
com.win.openPopup("/ui/SP/Common/Common_Popup01.xml", opt, data);
```

```xml
<!-- 팝업 XML - ref로 바인딩 -->
<w2:textbox id="titleTextbox" ref="data:opt.popupName" tagname="h3"></w2:textbox>
```

> **핵심**: WebSquare는 `id`가 아닌 `ref`로 데이터 바인딩. `id="opt.popupName"` 은 바인딩 아님

### 그리드 스크롤 처리
데이터가 늘어나도 팝업이 늘어나지 않고 그리드 안에서 스크롤 생기게 하려면:
```xml
<!-- visibleRowNum="all" 제거하고 숫자로 고정 -->
<w2:gridView style="height:354px;" visibleRowNum="10">
```

## 3. 보증 신청 업서트 로직 관련

### 케이스 구분

업서트(있으면 UPDATE, 없으면 INSERT) 관련 두 가지 케이스를 구분해야 함:

| 케이스 | 처리 방식 |
|--------|-----------|
| 신청하기 버튼 중복 클릭 | 이미 신청됐다는 알럿 표시 후 차단 |
| 승인 거절 후 재신청 | 기존 데이터 화면에 바인딩만, 저장 시 **새 INSERT** (새 신청번호 채번) |

거절 후 재신청을 UPDATE로 처리하면 기존 거절 이력이 덮어써짐.
금융권에서는 이력 보존이 중요하므로 새 INSERT가 원칙.

### 머지문 vs 프로그램 내 if/else

**회의에서 나온 주장**: 동시성(병합성) 문제 때문에 머지문보다 if/else 선호

**→ 이 이유는 사실 관계와 논리 모두 틀렸음**

#### 사실 관계 오류

`ON DUPLICATE KEY UPDATE`는 DB 엔진이 atomic하게 처리하는 단일 SQL 문이다.
반면 if/else는 SELECT → INSERT 사이에 gap이 존재해 다른 트랜잭션이 끼어들 수 있다(TOCTOU).

```
# if/else 방식 - gap 존재
SELECT (없음 확인)   ← 여기서 다른 트랜잭션 진입 가능
INSERT               ← 중복 발생 위험

# 머지문 - atomic
INSERT or UPDATE     ← DB 엔진이 한 번에 처리
```

즉, 동시성 측면에서는 머지문이 if/else보다 더 안전하다.

#### 논리 오류

동시성이 걱정된다면 결론은 "머지문을 써라"가 되어야 한다.
"동시성이 걱정되니 if/else를 쓰자"는 이유가 결론을 지지하지 못하는 논리 오류다.

#### 그럼 if/else를 선호하는 진짜 이유는?

동시성이 아니라 **비즈니스 로직 제어**가 실제 근거다:

1. **상태 조건 분기** — 거절/신청중 등 상태값에 따라 다른 처리가 필요한데, 머지문으로는 이를 SQL 한 줄로 표현하기 어려움
2. **이력 추적** — 재신청 시 새 채번이 필요한 경우 UPDATE로는 처리 불가
3. **가독성/유지보수** — 조건이 복잡해질수록 머지문은 읽기 어려움

```java
// 권장 방식 - 비즈니스 로직을 명확하게 제어
List result = selectById(id);
if (result.size() > 0) {
    // 상태에 따라 알럿 or 화면 바인딩
} else {
    insert(...); // 새 채번으로 INSERT
}
```

#### WRM 시스템에서 동시성 위험도

| 항목 | 내용 |
|------|------|
| 사용자 유형 | 내부 직원 (B2B) |
| 신청 주체 | 특정 직원 본인만 자기 권한 신청 가능 |
| 동시 신청 가능성 | 같은 사람이 동시에 두 탭에서 동시 클릭 |
| 트래픽 규모 | 대용량 B2C 아님 |

→ 같은 `EMP_NO`로 동시에 두 요청이 날아올 상황이 현실적으로 없으므로 실질적 위험은 거의 없음.

#### 동시성 방어가 필요하다면

if/else만으로는 완전하지 않다. 실무 표준은 **UK 제약조건 + 예외처리 병행**:

```java
try {
    insert(...);
} catch (DuplicateKeyException e) {
    // 중복 신청 알럿 처리
}
```

DB UK 제약조건이 최후 방어선 역할을 하므로, if/else와 조합하면 충분히 안전하다.

> **정리**: if/else를 쓰는 이유는 동시성 때문이 아니라 비즈니스 로직을 명확히 제어하기 위함.
> 동시성 문제 실질 위험은 이 시스템에서 무시해도 되는 수준이며, 필요 시 UK 제약 + 예외처리로 보강.
