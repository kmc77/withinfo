## 1. 인계 내용 파악

### selectNtnEsntlDsgnBrkd (지정/해지 이력 조회)

연도별 MAX PRHS_SN 1건 + 마지막 차수(MAX GIVE_DSPR) 1건만 조회하도록 수정.

```sql
AND A.NTN_ESNTL_GRSH_NO = #{ntnEsntlGrshNo}
AND A.PRHS_SN = (
    SELECT MAX(PRHS_SN)
    FROM TPS_NTN_ESNTL_GRSH_H A2

      AND SUBSTR(A2.DSGN_YMD, 1, 4) = SUBSTR(A.DSGN_YMD, 1, 4)

```

인계 시점: `when` 분기 완료, `otherwise` 분기 잔여로 기록되어 있었음.  
→ 실제 코드 확인 결과 **otherwise도 이미 적용 완료 상태**였음.

## 2. 회의록 분석 — UNION ALL 도입 배경

### 기존 구조
A(지급내역) + B(선대정보) 2개 조인만 존재. 이력 테이블 없음.

### 이력 테이블 생기면서 문제
국가필수선대 등록 페이지에서 지정/해지일자 수정 시 `TPS_NTN_ESNTL_GRSH_H`에 INSERT로 쌓임.  
→ B 테이블 값이 최신값이 아닐 수 있음 → H 테이블 최신값을 사용해야 함.

### 왜 단순 LEFT JOIN이 아닌 UNION ALL인가
단일 쿼리로 "이력 없으면 B값, 있으면 H 최신값"을 표현하면 복잡해짐.  
회의에서 결정: **기존 쿼리 그대로 위에 두고, 이력 조인한 쿼리를 아래에 UNION ALL로 붙여라.**

| 분기 | 조건 | 지정/해지일자 출처 |
|------|------|------|
| 1번 쿼리 | `NOT EXISTS` (이력 없는 선대) | B 테이블 |
| 2번 쿼리 | 이력 있는 선대 | H 테이블 MAX(PRHS_SN) 최신값 |

## 3. 쿼리 분기 구조

### 1단계: `<choose>` — 합계여부 체크박스

| 분기 | 조건 | 결과 |
|------|------|------|
| `when sumYn == "0"` | 체크 안함 | 차수별 전체 펼쳐서 표시 |
| `otherwise` | 체크함 | 연도별 합계로 묶어서 표시 |

### 2단계: 각 분기 안 UNION ALL

| 분기 | 조건 | 지정/해지일자 |
|------|------|------|
| 1번 쿼리 | `NOT EXISTS` | `B.DSGN_YMD`, `B.RMV_YMD` |
| 2번 쿼리 | 이력 있는 선대 | `H.DSGN_YMD`, `H.RMV_YMD` |

## 4. 문제 발견 — 사업연도 ↔ 지정일자 불일치

### 증상
2020년 사업연도 행에 2023년 지정일자가 표시됨.

### 원인
**1번 쿼리 (이력 없는 선대)**: B 테이블에 선대 1건당 DSGN_YMD가 1개인데, A 테이블은 같은 선대번호로 연도별 여러 행 존재. 선대번호로만 조인하면 B.DSGN_YMD=2023인 값이 2020, 2021, 2022, 2023 모든 연도 행에 붙음.

**2번 쿼리 (이력 있는 선대)**: MAX(PRHS_SN) 서브쿼리에 연도 조건이 없어서 전체 연도 중 MAX를 반환. 2020년 행에 2023년에 등록된 이력이 붙음.

## 5. 수정 내용

### 수정 1 — 1번 쿼리 WHERE절 추가

```sql
AND NOT EXISTS (
    SELECT 1 FROM TPS_NTN_ESNTL_GRSH_H HH
    WHERE HH.NTN_ESNTL_GRSH_NO = B.NTN_ESNTL_GRSH_NO
)
AND SUBSTR(B.DSGN_YMD, 1, 4) = A.BIZ_YR  -- 추가
```

→ 지정일자 연도가 사업연도와 같은 것만 필터링  
→ B 테이블이 연도별로 행이 따로 존재하는 구조이므로 **누락 없음**

### 수정 2 — 2번 쿼리 연도 조건 추가

```sql
AND SUBSTR(H.DSGN_YMD, 1, 4) = A.BIZ_YR  -- 추가
AND H.PRHS_SN = (
    SELECT MAX(PRHS_SN)
    FROM TPS_NTN_ESNTL_GRSH_H H2
    WHERE H2.NTN_ESNTL_GRSH_NO = B.NTN_ESNTL_GRSH_NO
    AND SUBSTR(H2.DSGN_YMD, 1, 4) = SUBSTR(H.DSGN_YMD, 1, 4)  -- 추가
)
```

→ 해당 연도 안에서만 최신 이력 1건 조회

### 적용 범위
* `when` 분기: 수정 완료 + 검증 완료
* `otherwise` 분기: 동일하게 적용 필요 (잔여 — 합계여부 체크박스 체크 시 에러 발생 중)

## 6. JDBC-28017 에러

### 증상
MAX 서브쿼리 안에 `A.BIZ_YR` 추가했더니 `JDBC-28017: Logical error in transformer` 발생.

### 원인
Altibase에서 MAX 서브쿼리 안에서 **외부 테이블(A) 직접 참조 불가**.

### 해결
`A.BIZ_YR` 대신 `SUBSTR(H.DSGN_YMD, 1, 4)` 로 대체.  
H는 이미 위에서 `SUBSTR(H.DSGN_YMD, 1, 4) = A.BIZ_YR`로 연도가 고정되어 있으므로 **결과 동일**.

---

## 7. MAX절 안에 SUBSTR이 필요한 이유

`H`에 연도 조건이 걸려있어도 서브쿼리 `H2`는 **독립적으로 전체 테이블 스캔**.

예: 320025D
- H: 2020년 조건 → PRHS_SN=1
- H2 연도 조건 없으면: 전체 중 MAX → PRHS_SN=5
- `H.PRHS_SN(1) = MAX(5)` → **불일치 → 결과 없음**

→ H2에도 반드시 `SUBSTR(H2.DSGN_YMD, 1, 4) = SUBSTR(H.DSGN_YMD, 1, 4)` 추가 필요.

## TIL

* **UNION ALL 도입 이유**: 이력 없는 선대(B값) / 있는 선대(H 최신값) 케이스를 단일 쿼리로 처리하기 복잡 → 케이스 분리 후 UNION ALL로 결합
* **WHERE절 필터링 vs 조인**: `SUBSTR(B.DSGN_YMD, 1, 4) = A.BIZ_YR`은 조인이 아닌 WHERE절 필터링 — 이미 선대번호로 조인된 상태에서 연도 일치 행만 걸러내는 것
* **MAX 서브쿼리 독립 실행**: 외부 테이블 조건이 걸려있어도 서브쿼리 내 별칭(H2)은 독립 실행 → 반드시 동일한 연도 조건 추가 필요
* **JDBC-28017**: Altibase에서 MAX 서브쿼리 안에서 외부 테이블(A) 직접 참조 불가 → 내부 참조(H, B)로 대체
* **차수별 전체 표시**: 사업년도별 지급내역은 차수 전부 보여주는 화면 — 같은 선대 여러 행은 차수가 다른 정상 데이터
* **B 테이블 구조**: 선대 1건당 DSGN_YMD 1개가 아닌 연도별로 행이 따로 존재 → `SUBSTR(B.DSGN_YMD, 1, 4) = A.BIZ_YR` 조건 추가해도 누락 없음
* **인계 하네스와 실제 코드 불일치 주의**: 인계서에 잔여 작업으로 기재되어 있어도 실제 코드 직접 확인 필수
