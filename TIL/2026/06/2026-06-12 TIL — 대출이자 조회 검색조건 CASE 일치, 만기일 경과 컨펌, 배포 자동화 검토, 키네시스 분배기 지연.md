## 2. 화면단 파일 배포 자동화 시도

### 배경
WebSquare 소스(`C:\KOBC_NGA\workspace\nga.ui\WebContent\...`) 수정 후
톰캣 배포 경로(`C:\KOBC_NGA\workspace\.metadata\.plugins\org.eclipse.wst.server.core\tmp0\wtpwebapps\nga.user\...`)에
**매번 수동 복사**하는 패턴.

이건 원래 Eclipse + WTP의 자동 publish가 처리하는 영역인데,
폐쇄망 SI 환경 특성상 수동 복사로 굳어진 듯.

### 검토한 방법들

| 방법 | 평가 |
|---|---|
| 터미널 cp 명령 | 화면마다 경로 달라서 함수 하드코딩 비효율 |
| 프로필 함수 (`function deploy`) | 오버 — 어차피 두 경로 다 입력해야 함 |
| 키네시스 매크로로 `dp ` 자동입력 | 4글자 줄이는 거 외엔 의미 없음 |
| **mklink 심볼릭 링크** | 한 번 설정하면 끝. 양쪽이 같은 실체. **다만 폴더 비어있어야 생성 가능** |
| **탐색기 "보내기" 메뉴 등록** | `shell:sendto`에 대상 폴더 바로가기 등록 → 우클릭 2번이면 끝 |
| Eclipse External Tools + 단축키 | WebSquare에서 편집하니 안 맞음 |

### 결론
- "터미널 + 매크로" 조합은 단계 수가 거의 안 줄어서 생산성 향상 미미
- 진짜 빠른 건 **mklink** 또는 **보내기 메뉴**
- 더 근본은 **Eclipse 자동 publish 설정 확인** (Server > Publishing > Automatically publish)

### 교훈
복사 단계를 줄이려면 "한 키 줄이기"가 아니라 **단계 자체를 없애야** 함.
매크로/별칭은 본질적인 단계 압축이 아니라 표면적 단축임.

## 3. 키네시스 분배기 전환 시 인식 지연

### 증상
Cable Matters 4-port USB 3.0 Switch에서 PC 전환 시 Kinesis 인식 늦음.
USB 전원 옵션 껐는데도 동일.

### 원인 (전원 아님)
- 분배기 전환 = USB **enumeration** 재발생 (장치 뽑았다 꽂은 것)
- Windows가 매번 드라이버 매칭 + descriptor 읽기 처음부터
- Kinesis는 **복합 장치**(키보드 + 마우스 + RGB + V-Drive) → 인식할 게 많음 → 느림
- "USB 선택적 절전 모드"는 절전 막는 거지 enumeration 자체를 못 막음

### "빠를 때가 있다" 단서
일관되게 느린 게 아니면 enumeration 외 변수 존재.

가능성:
1. **V-Drive 상태** — 켜짐 = Mass Storage까지 enumeration / 꺼짐 = HID만. **가장 유력**
2. **USB 드라이버 캐시** — 짧은 간격 재전환이면 빠르고, 오래 떨어졌다 돌아오면 느림
3. **포트 위치** — USB 3.0/2.0 협상 시간 차이
4. **대상 PC 상태** — 슬립/락이면 USB 스택 초기화 늦음

### 근본 해결
- EDID/USB emulation KVM 교체 (TrendNet TK209K, ConnectPro UR-12+)
- 또는 Kinesis 단일 PC 직결 + 보조 키보드

### V-Drive 토글
`SmartSet + F8` (메모리 기존 항목)

## 4. 대출이자 조회 진행상태 종료 검색 분기 누락

### 문제
화면에 "종료"로 표시되는 행이, 검색조건 진행상태에서 "종료" 선택 시 안 잡힘.

### 원인 구조

**Before 1 — 화면단 자동 변환 (주석처리됨)**
```javascript
// 만기일 < 오늘 이면 진행상태를 종료로 setCellData
/* for(var i = 0; i < cnt; i++) {
    var mtryYmd = dlt_loanIntSprtBizListVO.getCellData(i, "mtryYmd");
    if(today > mtryYmd) {
        dlt_loanIntSprtBizListVO.setCellData(i, "prgrsStClssCd", "31410002");
    }
} */
```

**Before 2 — 쿼리 SELECT의 CASE (현재 적용 중)**
```sql
CASE WHEN A.MTRY_YMD < TO_CHAR(SYSDATE,'YYYYMMDD')
      AND A.PRGRS_ST_CLSS_CD = '31410001'
     THEN '31410002'
     ELSE A.PRGRS_ST_CLSS_CD
 END AS PRGRS_ST_CLSS_CD
```

**문제:** WHERE절은 여전히 원본 컬럼 비교
```sql
AND A.PRGRS_ST_CLSS_CD = #{prgrsStClssCd}
```
→ DB상 값은 '31410001'이라 "종료" 필터에 안 잡힘.

### 해결 옵션 비교

| 옵션 | 장점 | 단점 |
|---|---|---|
| **A. WHERE에도 동일 CASE** | 간단, 의도 명확 | CASE 중복 (룰 변경 시 두 곳 동기화), 인덱스 못 탐 |
| B. 조건 분기 (`<if>`로 케이스별 WHERE) | 인덱스 활용 | 장황 |
| C. 인라인뷰로 감싸기 | CASE 한 곳만 관리 | 기존 쿼리 구조 흔듦 |

### 채택: A
```xml
<if test="prgrsStClssCd != null and prgrsStClssCd != ''">
  AND CASE WHEN A.MTRY_YMD <![CDATA[<]]> TO_CHAR(SYSDATE,'YYYYMMDD')
            AND A.PRGRS_ST_CLSS_CD = '31410001'
           THEN '31410002'
           ELSE A.PRGRS_ST_CLSS_CD
       END = #{prgrsStClssCd}
</if>
```

### A의 타당성 짚기
- 정석은 아님. CASE 중복 + 인덱스 X
- 더 근본은 **야간 배치로 PRGRS_ST_CLSS_CD를 실제 업데이트**하는 방식
  (대형 금융권 표준. 모든 화면/리포트 일관성 확보)
- 다만 이미 SELECT가 CASE로 가는 구조 → 일관성 측면 A가 맞음
- 데이터량 적고 성능 critical 아니면 A로 충분

### 화면단 ↔ 쿼리 변환 비교

| | 화면단 변환 (Before 1) | 쿼리 변환 (After) |
|---|---|---|
| 시점 | 조회 후 그리드 그릴 때 | DB에서 데이터 가져올 때 |
| 검색조건 일치 | ❌ | ✅ |
| 재사용 | ❌ (그 화면만) | ✅ (다른 화면/리포트) |
| 엑셀다운로드 반영 | ❌ | ✅ |

→ **화면 컨트롤 → 쿼리 컨트롤로 옮긴 것**

### 화면단 주석 정리

## 5. 만기일 경과 행 저장 시 컨펌 알림

### 요구사항
사용자가 상태/차수 변경 후 만기일이 현재보다 과거인 상태로 저장하려 하면 컨펌 알림.

### 기존 저장 구조 (SFPS_CSBS018M.xml, scwin.fn_save_click)
```javascript
scwin.fn_save_click = function() {
    if (dlt_loanIntSprtBizListVO.getTotalRow() == 0) { ... return; }
    if (com.util.isEmpty(dlt_loanIntSprtBizListVO.getModifiedIndex())) { ... return; }
    if (!scwin.fn_save_validation()) return false;

    com.win.message("QZZ0001", null, function(info) {
        if(info.click) com.sbm.execute(sbm_saveLoanIntSprtBiz);
    });
};
```

### 시도/실패 경과

**1차 시도 — 통합 CONFIRM**
`com.win._message(gcm.MESSAGE_TYPE.CONFIRM, "...", callback)`
→ 콜백 자체는 호출되는 듯하나 **확인 눌러도 sbm.execute가 트리거 안 됨**

**2차 시도 — `com.win.message`에 직접 텍스트 전달**
→ `com.win.message`는 메시지ID 패턴 → 직접 텍스트 미지원, 동작 X

**3차 시도 — `com.win.message("QZZ0001", "텍스트", callback)`**
→ 알림은 뜨는데 확인 시 저장 안 됨 (QZZ0001에 `${}` 치환자 없으면 콜백 안 탐 추정)

**4차 시도 — CONFIRM + return**
```javascript
if(/* expired */) {
    com.win._message(gcm.MESSAGE_TYPE.CONFIRM, "...", function(info) {
        if(info.click) com.sbm.execute(sbm_saveLoanIntSprtBiz);
    });
    return;
}
```
→ `com.win._message` 비동기 + CONFIRM 콜백 동작 불안정으로 실패

**5차 최종 — 알림만 띄우고 저장 중단**
사용자가 만기일 수정 후 다시 저장하는 흐름으로 변경.

### 최종 코드
```javascript
scwin.fn_save_click = function() {
    if (dlt_loanIntSprtBizListVO.getTotalRow() == 0) {
        com.win.message("EZZ0030", "저장할 데이터|");
        return;
    }

    if(com.util.isEmpty(dlt_loanIntSprtBizListVO.getModifiedIndex())) {
        com.win.message("IZZ0052");
        return;
    }

    if(!scwin.fn_save_validation()) {
        return false;
    }

    // 만기일 경과 행 차단
    var today = com.date.getServerDateTime().substring(0, 8);
    var rows = dlt_loanIntSprtBizListVO.getModifiedIndex();
    for(var i = 0; i < rows.length; i++) {
        if(dlt_loanIntSprtBizListVO.getCellData(rows[i], "mtryYmd") < today) {
            com.win._message(gcm.MESSAGE_TYPE.INFO, "만기일이 경과한 행이 있습니다.");
            return;
        }
    }

    //저장하시겠습니까?
    com.win.message("QZZ0001", null, function(info) {
        if(info.click) {
            com.sbm.execute(sbm_saveLoanIntSprtBiz);
        }
    });
};
```

### 핵심 학습

| 함수 | 동작 |
|---|---|
| `getModifiedIndex()` | 수정/추가된 행 인덱스 배열 반환 (KOBC 컨벤션 확정) |
| `com.win.message(msgId, param, callback)` | **메시지ID 전용**. 직접 텍스트 X. `${}` 치환자 위치에 param 들어감 |
| `com.win._message(TYPE, text, callback)` | 직접 텍스트 가능. **INFO는 동작 검증**, **CONFIRM 콜백은 불안정** |
| `gcm.MESSAGE_TYPE.INFO` | 단순 알림 — 안정적 |
| `gcm.MESSAGE_TYPE.CONFIRM` | 확인/취소 — 콜백 시그니처 명확하지 않음, 화면별 차이 있을 수 있음 |

### 못 풀고 남긴 것
`com.win._message` CONFIRM이 왜 동작 안 했는지 정확한 원인 미확인.
- 콜백 시그니처가 `function(info)` / `function(ok)` / `function()` 중 무엇인지
- `info.click` 외 다른 프로퍼티명일 가능성
- → 다음 세션에서 `console.log(info)` 찍어서 확인 권장

### 일관성 vs 동작
브라우저 기본 `confirm()`은 동작은 하지만 **퍼블 일관성 깨짐**.
회사 공통 confirm 패턴을 못 쓴다면 → 알림 + 사용자 재저장이 차선.
정석은 **신규 메시지ID 등록** (사수 컨펌 후).
